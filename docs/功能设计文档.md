# 天织 Ontop UI 功能设计文档

## 1. 文档目的

本文档用于说明 `ontop-ui` 当前版本的功能范围、目标用户、页面设计、交互流程、接口依赖和功能约束，作为产品设计、研发实现和测试验收的共同依据。

文档基于当前代码和页面实现整理，不包含尚未落地的规划功能。

## 2. 产品定位

天织 Ontop UI 是一个面向语义建模和虚拟知识图谱管理的工作台，目标是围绕 Ontop 提供一条完整操作链路：

- 接入数据库
- 理解数据库结构
- 自动生成本体与映射
- 调整映射规则
- 执行 SPARQL
- 通过 AI 做自然语言查询
- 以图谱和定义视角查看本体

## 3. 目标用户

当前版本主要面向以下用户：

- 数据建模人员
- 语义层开发人员
- 数据平台工程师
- 本体原型验证人员
- 演示与售前方案人员

## 4. 功能范围

当前版本已实现模块：

- 首页
- 登录页
- 数据源管理
- 数据库概览
- 映射编辑
- SPARQL 查询
- AI 助手
- 本体可视化
- AI 设置
- 系统设置
- 语义注释
- 业务词汇表
- 本体精化建议
- 端点注册表（后台功能）
- API Key 鉴权（中间件）

不在当前版本范围内：

- 多用户权限控制
- 正式账号认证
- 工作流审批
- 文件版本回滚
- 完整审计日志页面
- 多租户隔离

## 5. 用户角色与权限假设

当前版本默认单一角色视角：

- 管理员 / 建模者

行为特征：

- 可以访问所有页面
- 可以创建删除数据源
- 可以修改映射和 AI 配置
- 可以重启 Ontop Endpoint

说明：

- 当前代码中没有真实权限控制
- 登录页和系统页展示的用户信息是 mock 数据

## 6. 用户任务主线

核心任务主线如下：

1. 配置数据源
2. 测试连接并读取 schema
3. 执行全量或局部 Bootstrap
4. 查看和修改映射规则
5. 验证映射并启动端点
6. 执行 SPARQL 查询
7. 配置 AI 模型
8. 用自然语言提问
9. 查看本体图谱和定义

### 7.11 语义注释 `/annotations`

#### 目标

为自动生成的本体提供人工可控的语义标注层，将中英文标签/注释与 TTL 文件解耦存储。

#### 功能点

- 统计卡片：待审核/已接受/已拒绝/合计
- 待审核列表：LLM 生成的标注，可逐条接受/拒绝/编辑
- 已接受列表：将合并入 active TTL
- 已拒绝列表：不写入 TTL
- 搜索过滤：实体名、类型（类/数据属性/对象属性）
- 批量操作：全选/全部接受/全部拒绝
- 编辑对话框：修改中英文名称和说明
- 合并到本体：将 accepted 注释合并到 active TTL

#### 人工标注优先级

| 操作 | source | status | Bootstrap 重跑后 |
|------|--------|--------|----------------|
| LLM 自动生成 | llm | pending | 被替换 |
| 用户接受 | llm | accepted | 永久保留 |
| 用户手动编辑 | human | accepted | 永久保留 |
| 用户拒绝 | llm | rejected | 永久保留 |

#### 依赖接口

- `GET /api/v1/annotations/{ds_id}`
- `GET /api/v1/annotations/{ds_id}/stats`
- `POST /api/v1/annotations/{ds_id}`
- `PUT /api/v1/annotations/{ds_id}/{id}`
- `DELETE /api/v1/annotations/{ds_id}/{id}`
- `POST /api/v1/annotations/{ds_id}/batch-status`
- `POST /api/v1/annotations/{ds_id}/merge`

### 7.12 业务词汇表 `/glossary`

#### 目标

将业务口语词显式映射到本体属性/类，消除 AI 查询时 LLM 猜测属性名导致的失败。

#### 功能点

- 统计卡片：全部/LLM 生成/人工添加
- 词汇表格：主词、别名、映射 URI、来源
- 自动生成：从已 accepted 注释调用 LLM 推导业务词汇
- 新增/编辑对话框
- 导出/导入 JSON
- 模糊搜索 + 类型过滤

#### 注入机制

用户提问时，按关键词命中打分（主词=3分、别名=2分、URI=1分），取得分最高的前 12 条注入 SPARQL 生成 Prompt。

#### 依赖接口

- `GET /api/v1/glossary/{ds_id}`
- `GET /api/v1/glossary/{ds_id}/stats`
- `POST /api/v1/glossary/{ds_id}`
- `PUT /api/v1/glossary/{ds_id}/{id}`
- `DELETE /api/v1/glossary/{ds_id}/{id}`
- `POST /api/v1/glossary/{ds_id}/generate`
- `GET /api/v1/glossary/{ds_id}/export`
- `POST /api/v1/glossary/{ds_id}/import`

### 7.13 本体精化建议 `/refinement`

#### 目标

通过 LLM 分析本体结构和已审核注释，给出结构化改进建议并支持一键应用。

#### 功能点

- 选择数据源后触发 AI 分析（后台任务）
- 建议列表按优先级排序
- 逐条接受/拒绝
- 单条应用或批量应用
- TTL 修改前自动备份

#### 建议类型与风险分级

| 类型 | 含义 | 自动应用 |
|------|------|----------|
| `RENAME_CLASS` | 重命名类 | 自动 |
| `RENAME_PROPERTY` | 重命名属性 | 自动 |
| `REFINE_TYPE` | 精化 XSD 数据类型 | 自动 |
| `ADD_LABEL` | 补充中文标注 | 自动 |
| `ADD_SUBCLASS` | 添加子类关系 | 人工 |
| `MERGE_CLASS` | 合并冗余类 | 人工 |

#### 依赖接口

- `POST /api/v1/suggestions/{ds_id}/analyze`
- `GET /api/v1/suggestions/{ds_id}`
- `GET /api/v1/suggestions/{ds_id}/stats`
- `PUT /api/v1/suggestions/{ds_id}/{id}/status`
- `POST /api/v1/suggestions/{ds_id}/{id}/apply`
- `POST /api/v1/suggestions/{ds_id}/batch-apply`

### 7.14 端点注册表（后台功能）

#### 目标

多数据源端点信息持久化，支持一键切换激活数据源。

#### 功能点

- Bootstrap 完成后自动注册数据源信息
- 列出所有已注册数据源
- 获取当前激活数据源
- 切换激活数据源（文件同步 + Ontop reload）

#### 切换流程

1. `PUT /api/v1/endpoint-registry/{ds_id}/activate`
2. 文件同步到共享 active 目录
3. 调用 Ontop Admin API reload
4. 更新 is_current 标记

切换期间约 5-10 秒，SPARQL 请求返回 503。

#### 依赖接口

- `GET /api/v1/endpoint-registry`
- `GET /api/v1/endpoint-registry/current`
- `PUT /api/v1/endpoint-registry/{ds_id}/activate`

### 7.15 API Key 鉴权（中间件）

#### 目标

保护对外暴露的 API 和 MCP 端点，防止未授权访问。

#### 鉴权策略

| 请求来源 | 鉴权方式 |
|----------|----------|
| 前端（X-Internal-Request 头） | 放行 |
| localhost（127.0.0.1/::1） | 放行 |
| api_enabled=false | 放行 |
| 其他外部请求 | 校验 X-API-Key 头或 api_key 参数 |

#### 保护范围

- `/api/v1/*` 路由
- `/mcp/*` 路由（MCP Server）

#### 依赖接口

- `POST /api/v1/publishing/api/generate-key`

## 8. 功能流程设计

### 8.1 主流程一：从数据库到本体

1. 在数据源管理页创建数据源
2. 测试连接
3. 读取 schema
4. 在数据库概览页选择全量或局部范围
5. 生成本体和映射
6. 在映射编辑页修正规则

### 8.2 主流程二：从映射到查询

1. 在映射编辑页验证映射
2. 重启 Endpoint
3. 在 SPARQL 页执行查询
4. 查看 SQL 重写
5. 确认结果正确

### 8.3 主流程三：从查询到 AI 问答

1. 在 AI 设置页配置模型
2. 在 AI 助手页输入自然语言问题
3. 查看自动生成的 SPARQL
4. 查看 SQL 和结果
5. 获取最终回答

### 8.4 主流程四：语义增强（注释 + 词汇表 + 精化）

1. Bootstrap 后 LLM 自动生成 pending 语义标注
2. 在 `/annotations` 审核（接受/编辑/拒绝）
3. 批量接受后点击「合并到本体」
4. 在 `/glossary` 自动生成业务词汇，或手工添加关键术语
5. 在 `/refinement` 触发 AI 分析，审核并应用精化建议
6. 返回 AI 助手验证语义增强效果

### 8.5 主流程五：多数据源切换

1. 对多个数据源分别执行 Bootstrap（自动注册到端点注册表）
2. 通过 `PUT /endpoint-registry/{ds_id}/activate` 切换激活数据源
3. 等待端点重启完成（5-10 秒）
4. 在 SPARQL/AI 助手查询当前激活数据源

## 10. 功能边界与约束

### 10.1 当前约束

- 仅支持单实例工作台模式
- 无真实用户体系（登录页为 mock）
- 无并发任务队列（后台任务通过 FastAPI BackgroundTasks 串行执行）
- 无文件版本回滚界面
- AI 助手依赖外部 LLM 服务可用
- SPARQL 与 AI 查询依赖 Endpoint 已启动
- 端点切换同一时刻只有一个 SPARQL 端点在服务
- 本体精化 REFINE_TYPE 使用正则匹配，极少数情况可能误替换
- LLM JSON 输出可能包含未闭合引号，已内置 repair 机制兜底

### 10.2 使用前提

- Ontop CLI 路径正确
- 输出目录可写
- 目标数据库可连接
- 对应 JDBC 驱动可用
- LLM 服务地址正确

## 11. 异常场景设计

### 11.1 数据源异常

场景：

- JDBC URL 错误
- 用户名密码错误
- 数据库不可达

处理：

- 页面 toast 报错
- 后端返回连接失败消息

### 11.2 Bootstrap 异常

场景：

- Ontop CLI 执行失败
- 输出目录权限不足
- 选定表不存在或有歧义

处理：

- 后端返回 400
- 页面显示失败信息

### 11.3 查询异常

场景：

- Endpoint 未启动
- SPARQL 语法错误
- AI 生成了无效 SPARQL

处理：

- SPARQL 页面显示错误文本
- AI 助手显示失败消息

### 11.4 AI 配置异常

场景：

- 模型发现失败
- API Key 错误
- Base URL 无法访问

处理：

- 设置页回退到预设模型或手工输入
- 保存不阻断页面继续编辑
