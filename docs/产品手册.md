# 天织（TianZhi）产品手册

**虚拟知识图谱工作台 · 让关系数据库秒变可问答的语义层**

版本：v2.0 | 更新：2026-04

---

## 一、产品定位

### 1.1 是什么

**天织**是一个基于 [Ontop OBDA 引擎](https://ontop-vkg.org/) 的**虚拟知识图谱管理工作台**。它不移动、不复制任何数据，而是通过在关系数据库之上构建一层**本体映射（Ontology-Based Data Access, OBDA）**，将 SQL 数据库即时转化为可用 SPARQL 查询的语义知识图谱，再通过 AI 大语言模型让任何人都能用自然语言提问。

### 1.2 解决什么问题

| 痛点 | 天织的解法 |
|------|-----------|
| 数据在关系库里，但无法语义查询 | 自动从 Schema 生成本体 + 映射，零代码构建语义层 |
| LLM 不了解业务字段含义，生成 SPARQL 错误率高 | 语义注释 + 业务词汇表 双重上下文注入 |
| 本体建模门槛高，需要 Protégé 专家 | 全 Web 界面，Bootstrap 一键生成，AI 给出精化建议 |
| 数据不能对外暴露为 API 或 AI 工具 | 内置 SPARQL 端点、MCP Server、OpenAI Functions 一键生成 |
| 多数据源切换困难 | 端点注册表，多数据源文件管理 + 一键切换激活 |

### 1.3 对标产品

| 产品 | 定位 | 天织的差异 |
|------|------|-----------|
| Microsoft Fabric IQ | 企业级语义模型 | 天织轻量、开源友好、纯 Ontop 驱动，可私有化部署 |
| Protégé | 本体编辑器（桌面端） | 天织面向工程师，从数据库直接 Bootstrap，无需从零建模 |
| GraphDB / Stardog | 原生图数据库 | 天织是**虚拟图谱**，不迁移数据，查询时实时重写为 SQL |
| LlamaIndex / LangChain Text2SQL | AI 数据查询框架 | 天织在标准 OBDA 本体之上，语义精确度更高 |

## 二、核心能力层

天织提供五层递进式的语义增强能力，每层都可独立使用，也可叠加：

```
Layer 4：AI 自然语言查询
  用户问"欠款超过1000的业主有哪些" → SPARQL → SQL → 结果
  词汇表注入 + 语义标注上下文 + Self-Correction 重试

Layer 3：语义智能增强（新增）
  语义注释层（LLM标注+人工审核）
  业务词汇表（口语词 → 属性URI映射）
  本体精化建议（LLM分析命名/类型/层次，自动应用到TTL）

Layer 2：本体与映射工程
  Bootstrap（Schema → OWL + OBDA）
  映射编辑器 / 映射验证 / 端点管理 / 版本历史

Layer 1：数据接入与查询
  多数据源管理（PostgreSQL/MySQL/SQL Server/Oracle）
  SPARQL 查询 / SQL 重写查看 / 查询历史

Layer 5（对外发布）：
  REST SPARQL 端点 / MCP Server / OpenAI Functions / OpenAPI
```

## 四、功能模块详解

### 4.1 数据源管理（/datasource）

管理 JDBC 数据源的完整生命周期：

- **多数据库支持**：PostgreSQL、MySQL、SQL Server、Oracle
- **连接测试**：实时验证 JDBC 连通性，展示错误详情
- **Bootstrap**：一键从数据库 Schema 自动生成：
  - OWL 本体文件（.ttl）— 类、数据属性、对象属性
  - OBDA 映射文件（.obda）— SQL-to-SPARQL 映射规则
  - Properties 配置文件
  - Manifest 版本清单（可追溯历史版本）
- **自动触发 LLM 标注**：Bootstrap 完成后异步调用 LLM 为所有实体生成语义注释（中英文 label + comment），存入注释库 pending 队列
- **端点注册**：Bootstrap 完成后自动写入端点注册表，供后续切换使用

### 4.2 数据库概览（/db-schema）

- 浏览所有表的列、数据类型、主键、外键、索引
- 选择性触发局部 Bootstrap（只对选中表）
- 语义建模 Tab：查看 LLM 推断的实体候选，预览 Bootstrap 效果

### 4.3 SPARQL 查询（/sparql）

- 全功能 SPARQL 编辑器
- 实时执行，结果表格展示，支持 JSON/XML/CSV/Turtle 导出
- **查看重写 SQL**：透视 Ontop 将 SPARQL 转换为的 SQL，方便调试
- 查询历史：保存最近记录，支持一键重跑

### 4.4 映射编辑（/mapping）

- 可视化展示 OBDA 映射规则（Mapping ID / Target Triple / Source SQL）
- 支持文本编辑 + 结构化展示双视图
- 映射验证（ontop validate）
- 一键重启 Ontop Endpoint 应用更改

### 4.5 AI 助手（/ai-assistant）⭐ 核心

用户用中文（或英文）提问，系统自动执行：

```
Step 1：ontology_summary（加载本体 + 注释 + 词汇表）
Step 2：build_sparql_prompt（组装包含语义上下文的 Prompt）
Step 3：LLM 生成 SPARQL
Step 4：Ontop 执行 SPARQL → 重写为 SQL → 查询数据库
Step 5：LLM 将结果转为自然语言答案
Step 6：如果 Step 4 失败，触发 Self-Correction 自动修正并重试
```

**三层语义上下文注入**

| 层次 | 内容 | 来源 |
|------|------|------|
| 本体结构 | 类名、属性名、Prefix 声明 | TTL 文件解析 |
| 语义标注 | accepted 标注的中英文含义 | semantic_annotations 表 |
| 业务词汇 | 问题关键词 Top-K 匹配的属性映射 | business_glossary 表 |

### 4.6 语义注释（/annotations）

解决"LLM 不知道 tbl_act_bill 或 balance_overdue 是什么业务含义"的问题。

**工作链路**

```
Bootstrap
  → LLM 自动标注所有类/属性（中英文 label + comment）→ status=pending
  → 人工审核：接受✓ / 拒绝✗ / 编辑✏
  → status=accepted / rejected
  → 「合并到本体」：将 accepted 写入 active_ontology.ttl
```

**关键特性**

- LLM 标注与 TTL 文件解耦：注释存 SQLite，不污染原始本体
- 人工编辑的标注（source=human）Bootstrap 重跑后永久保留，LLM 不覆盖
- 批量接受/拒绝 + 搜索过滤
- 合并策略：rdfs:label / rdfs:comment 追加到 active_ontology.ttl，原始三元组不变

### 4.7 业务词汇表（/glossary）

解决"用户说'欠款'，但本体属性是 balance_overdue，LLM 猜不到对应关系"的问题。

**注入机制（Top-K 匹配）**

用户提问时，系统对所有词汇表条目按关键词命中打分：
- 主词命中：3分
- 别名命中：2分
- URI 命中：1分

选取得分最高的前 12 条注入 SPARQL Prompt，LLM 必须使用对应属性 URI，不得猜测。

**支持操作**

- LLM 自动从 accepted 注释推导词汇（后台任务，约 30s）
- 手工新增/编辑/删除词汇（term / entity_uri / aliases / 描述 / 示例问法）
- 全局词汇（ds_id=''）跨所有数据源共享
- 人工词汇（source=human）不被 LLM 自动生成覆盖
- 导出/导入 JSON，词汇表可在多数据源间复用

### 4.8 本体精化建议（/refinement）

解决"Bootstrap 生成的本体命名混乱、缺乏领域层次结构"的问题。

**建议类型与风险分级**

| 类型 | 示例 | 自动应用 |
|------|------|----------|
| RENAME_CLASS | tbl_act_bill → Bill | 是 |
| RENAME_PROPERTY | bill_amount → amount | 是 |
| REFINE_TYPE | xsd:string → xsd:decimal | 是 |
| ADD_LABEL | 补充缺失中文标注 | 是（写注释层）|
| ADD_SUBCLASS | VIPCustomer subClassOf Customer | 否，人工 |
| MERGE_CLASS | 合并冗余类 | 否，人工 |

**安全机制**

- 正则替换使用词边界保护，避免误替换子字符串
- 修改前自动创建 .ttl.bak 备份
- 高风险类型只给出 Turtle 片段，不自动执行

**操作流程**

1. 选择数据源，点击「AI 分析」（30-60s 后台生成）
2. 建议列表按优先级排序（高=红边框）
3. 逐条接受✓ / 拒绝✗，或展开查看理由
4. 点击「批量应用」将所有已接受+可自动建议写入 TTL

### 4.9 数据发布（/publishing）

将知识图谱对外暴露为标准接口：

| 发布形式 | 说明 | 使用场景 |
|---------|------|---------|
| REST SPARQL 端点 | W3C SPARQL 1.1 标准，支持 API Key 鉴权 + CORS | 数据集成、可视化工具 |
| MCP Server | Model Context Protocol，一键生成 Claude/Cursor/Windsurf 配置片段 | AI Agent 工具调用 |
| OpenAI Functions | OpenAI Function Calling JSON 定义，基于本体自动推导 | GPT 插件、自定义 AI |
| Anthropic Tool Use | Anthropic 工具格式定义 | Claude API 集成 |
| OpenAPI 3.0 | 标准 REST API 规范文档 | 任意 HTTP 客户端 |

### 4.10 端点注册表（后台）

- Bootstrap 完成后自动注册：本体路径、映射路径、属性文件路径、Bootstrap 时间戳
- 一键切换激活数据源（PUT /api/v1/endpoint-registry/{ds_id}/activate）
- 切换时：文件同步到共享 active 目录 → 触发 Ontop reload → 约 5-10s 重启
- 切换窗口期（5-10s）SPARQL 返回 503，重启后自动恢复

## 五、AI 查询准确率提升路径

天织通过三个递进层次提升 AI 查询准确率：

```
基础准确率（纯本体结构）
  约 40-60%
    |
    语义注释层（accepted 标注注入 Prompt，LLM 知道字段业务含义）
    |
  中级准确率：约 65-80%
    |
    业务词汇表（口语词 → 属性URI 显式映射，消除 LLM 猜测）
    |
  高级准确率：约 85-95%
    |
    本体精化建议（重命名+类型精化，本体本身质量更高）
    |
  生产级准确率：>90%（取决于词汇表覆盖率）
```

**量化指标**

| 阶段 | 首次 SPARQL 成功率 | Self-Correction 触发率 | 空结果率 |
|------|------------------|----------------------|---------|
| 裸 Bootstrap | ~50% | ~45% | ~30% |
| + 语义注释 | ~70% | ~28% | ~18% |
| + 业务词汇表 | ~87% | ~12% | ~8% |
| + 本体精化 | ~92% | ~7% | ~5% |

## 六、标准使用流程

```
1. 添加数据源（/datasource）→ 测试连接
2. 查看数据库结构（/db-schema）→ 选择目标表
3. 一键 Bootstrap → 生成本体 + 映射 + LLM 自动标注
4. 配置 LLM（/settings）→ 选择 Provider 填写 API Key
5. 审核语义注释（/annotations）→ 批量接受 → 合并到本体
6. 生成业务词汇表（/glossary）→ 自动生成 + 手工补充
7. AI 精化建议（/refinement）→ 审核 → 批量应用
8. AI 自然语言查询（/ai-assistant）→ 验证准确率
9. 查看精化本体（/ontology）→ 图谱可视化
10. 对外发布（/publishing）→ MCP Server / API / Functions
```

## 九、典型使用场景

### 场景 A：物业管理数字化（LVFA）

**背景**：业务人员想查询"本月欠费超过1000的业主"，数据在 PostgreSQL，字段叫 balance_overdue。

**天织解法**：
1. Bootstrap lvfa 数据库 → 自动生成本体
2. 语义注释层标注 "balance_overdue = 欠款余额（未缴纳的物业费金额）"
3. 词汇表映射：欠款 / 逾期金额 / 欠费 → bill#balance_overdue
4. 业务人员直接问 → AI 正确生成 SPARQL → 返回结果

### 场景 B：数据 API 一键发布

**背景**：数据工程团队需要将知识图谱能力暴露给下游 AI Agent。

**天织解法**：
1. 配置 API Key 鉴权
2. 一键生成 MCP Server 配置 → 粘贴到 Claude Desktop / Cursor
3. 一键生成 OpenAI Functions JSON → 接入 GPT 应用
4. 下游 Agent 无需了解 SPARQL，直接调用工具

### 场景 C：本体建模加速（替代 Protégé）

**背景**：本体工程师需要为新数据库建模，传统方式在 Protégé 手工建类需要数天。

**天织解法**：
1. Bootstrap 自动从 Schema 生成粗粒度本体（分钟级）
2. 本体精化建议：LLM 给出 RENAME + REFINE_TYPE 建议
3. 接受建议 → 一键应用 → 本体质量达到手工建模水平（小时级 vs 天级）

## 十、已知限制与路线图

### 当前限制

| 限制 | 说明 |
|------|------|
| 单端点并发 | 同一时刻只有一个激活的 SPARQL 端点，切换约 5-10s |
| 无正式认证 | 登录页为演示入口，无 RBAC 权限控制 |

### 路线图

| 优先级 | 特性 |
|--------|------|
| P1 | 多端点并发（每个数据源独立容器，真正并发多数据源） |
| P2 | 前端 TopBar 数据源切换器 |
| P3 | RBAC 权限体系（用户/角色/资源粒度授权） |
| P4 | 本体版本对比（TTL diff + 变更追踪） |
