# 天织（TianZhi）
## 让数据"开口说话"的虚拟知识图谱平台

> **天织不移动你的数据，却让你的数据能被任何人用自然语言提问。**

---

## 天织是什么

天织是一个**虚拟知识图谱管理工作台**，基于国际标准 OBDA（Ontology-Based Data Access）技术，将关系数据库即时转化为可用自然语言查询的语义知识图谱。

```
你的 PostgreSQL / MySQL / SQL Server / Oracle
              ↓ 织一层语义（本体 + 映射）
        知识图谱（SPARQL 可查询）
              ↓ AI 自然语言理解
业务人员用中文提问 → 得到准确答案
              ↓ 标准接口发布
AI Agent / 其他系统 / 可视化平台
```

**核心承诺**：
- ⚡ **不移动数据**：查询时实时重写为 SQL，数据永远在你的库里
-  **不依赖云厂商**：私有化部署，LLM 支持本地模型
-  **不猜字段含义**：业务词汇表机制确保 AI 精准理解你的业务语言

## 三个层次的价值

### 层次一：让数据可以被语义查询

传统数据库只能靠 SQL 访问，天织在其上构建一层**本体映射（OWL + OBDA）**，让同样的数据支持：
- SPARQL 语义查询（国际 W3C 标准）
- 自然语言问答（中文/英文）
- 图谱可视化分析
- 跨系统语义关联

**你得到的**：同一套数据，多种访问维度，无需重复建仓。

---

### 层次二：让 AI 真正理解你的业务

AI 查询最大的失败原因不是模型不够聪明，而是**模型不了解你的业务**。

天织提供三层语义上下文，持续"教"AI 认识你的业务：

| 机制 | 解决的问题 | 效果 |
|------|-----------|------|
| **语义注释层** | LLM 不知道 `tbl_act_bill` 是什么 | 自动标注 + 人工审核，AI 知道每个字段的业务含义 |
| **业务词汇表** | 用户说"欠款"，AI 找不到 `balance_overdue` | 口语词 → 本体属性的显式映射，消除猜测 |
| **本体精化建议** | 字段命名混乱，本体结构扁平 | LLM 分析并建议重命名、类型精化、层次优化 |

**你得到的**：AI 查询准确率从 40-60% 提升至 90% 以上，业务人员提问不再需要"迎合系统"。

---

### 层次三：让数据成为可发布的 AI 能力

数据不只是给内部用的。天织将知识图谱一键发布为标准接口：

| 发布形式 | 使用场景 |
|---------|---------|
| **REST SPARQL 端点** | 标准语义数据 API，支持 API Key 鉴权 |
| **MCP Server** | Claude Desktop / Cursor / Windsurf 等 AI Agent 直接调用 |
| **OpenAI Functions** | 接入 GPT 及任意 Function Calling 生态 |
| **OpenAPI 3.0** | 标准 REST 文档，任意 HTTP 客户端可接入 |

**你得到的**：你的业务数据，变成每一个 AI Agent 都能调用的工具能力。

## 五大核心能力

### 一键 Bootstrap — 分钟级建本体

忘掉 Protégé，忘掉手工建模。

天织连接你的数据库，一键自动：
- 分析表结构、字段类型、外键关系
- 生成标准 OWL 本体（类、属性、关系）
- 生成 OBDA 映射规则（SQL → SPARQL 的转换逻辑）
- 触发 LLM 为所有类和属性自动标注业务语义

**效果**：原来需要本体专家数天的工作，现在数分钟完成。精化和审核只需要业务人员参与，不需要技术专家。

---

### 语义注释审核 — 让 AI 说"你的话"

Bootstrap 生成的本体有了结构，但 AI 还不知道业务含义。

天织的语义注释系统：
1. Bootstrap 完成后，LLM 自动为每个类和属性生成中英文标注（后台进行，不阻断流程）
2. 业务人员在审核界面逐条确认或修改（批量操作，通常 < 30 分钟）
3. 人工修改的标注永久保留，系统重建不丢失
4. 一键"合并到本体"，AI 立即获得新的语义上下文

**效果**：AI 从"猜字段"变为"懂业务"，首次生成 SPARQL 的成功率大幅提升。

---

### 业务词汇表 — 精准的"翻译官"

用户说"本月逾期业主"，数据库里是 `payment_overdue_days > 0`。

天织内置关键词匹配引擎：
- 维护"业务口语词 → 本体属性 URI"的映射字典
- 每次 AI 查询时，自动从词汇表中匹配最相关的前 12 条注入 Prompt
- LLM 收到"欠款 = bill#balance_overdue"的明确指引，不再猜测
- 词汇表可导出复用，跨数据源共享全局词汇

**效果**：彻底消除因字段命名差异导致的 AI 查询失败，AI 说的和业务人员想的是同一件事。

---

### 本体精化建议 — AI 优化 AI 生成的本体

Bootstrap 自动生成的本体往往带有数据库痕迹：表名前缀（`tbl_`）、冗余字段名、缺少层次结构。

天织的精化建议引擎：
- LLM 分析本体结构，给出优先级分组的改进建议
- 低风险改动（重命名、类型精化、补充标注）一键自动应用
- 应用前自动备份，可安全回滚
- 高风险改动（添加子类关系）给出操作指引，人工确认后执行

**效果**：本体质量从"数据库平铺结构"提升为"领域语义模型"，AI 查询准确率进一步提升。

---

### 多协议数据发布 — 一次建模，多处使用

知识图谱建好了，天织帮你把它变成可以复用的能力：
- **内部BI团队**：通过 SPARQL 端点接入 BI 工具

## 典型应用场景

### 物业管理数字化

**痛点**：业主欠费查询依赖 IT 出报表，等待周期长，业务人员无法自助查询。

**天织方案**：
- 连接物业管理系统数据库（PostgreSQL）
- Bootstrap 生成本体，标注"欠款余额 = balance_overdue"
- 词汇表建立"逾期 / 欠费 / 未缴"等口语词的对应关系
- 业务人员直接在聊天界面问："上月欠费超过500元、且超过90天的业主名单"
- 秒级返回结果，无需 SQL，无需报表申请

**价值**：数据自助查询，减少 IT 响应需求，业务响应速度从天级到秒级。

---

### 金融风控数据洞察

**痛点**：风控人员需要跨多张表关联分析客户风险，每次依赖数据工程师开发查询。

**天织方案**：
- 多表关系通过 OBDA 映射自动处理
- 风控分析师用自然语言描述查询条件
- 天织自动生成 SPARQL，Ontop 实时重写为多表 JOIN SQL
- 结果直接返回，无需等待技术支持

**价值**：风控响应速度提升，分析师可自主探索数据，IT 部门从重复开发中解放。

---

### 工业设备数据语义化

**痛点**：工厂设备传感器数据存在多套系统，标准不统一，无法统一查询。

**天织方案**：
- 为每个系统各自 Bootstrap 建立本体
- 通过统一的本体层对齐语义（同样是"设备温度"，不同系统字段名不同）
- AI 助手统一入口，跨系统自然语言查询
- MCP Server 接入 Cursor，工程师直接对话数据

**价值**：跨系统数据统一访问，无需重建数据中台，保留原有系统投资。

## 与现有系统如何集成

天织以**非侵入式**方式接入现有数据架构：

```
现有数据库（保持不变）
      |
      | JDBC 只读连接
      |
    天织
      |
      +── SPARQL 端点 ──── BI 工具 / 可视化平台
      |
      +── MCP Server ───── Claude / Cursor / Windsurf
      |
      +── REST API ──────── 自研应用 / 小程序 / 移动端
      |
      +── Function JSON ─── GPT 应用 / Coze / Dify
```

- **零侵入**：不需要在现有数据库安装任何插件或改变任何表结构
- **只读为主**：天织默认以只读方式访问数据库，不产生任何写操作
- **标准协议**：所有接口使用 W3C / OpenAI / Anthropic 标准格式，无厂商锁定
