# 天织 Ontop UI 使用说明书

本文档基于仓库当前代码整理，覆盖 `ontop-ui` 的真实运行方式、功能模块、操作流程、接口能力和已知限制。

## 1. 系统概述

`ontop-ui` 是一个基于 Ontop 的虚拟知识图谱工作台，目标是把关系数据库通过 Ontop 映射为可查询的语义层，并提供以下能力：

- 数据源管理
- 数据库结构探查
- Ontop Bootstrap 生成本体与映射
- OBDA 映射文件编辑
- SPARQL 查询与 SQL 重写查看
- AI 自然语言问答
- 本体可视化与 TTL 解析
- **语义注释层：LLM 自动标注 + 人工审核 + 合并到 active TTL**
- **业务词汇表：业务口语词 → 本体属性/类 的显式映射，注入 AI 查询 Prompt**
- **端点注册表：多数据源端点信息持久化，支持一键切换激活数据源**
- **本体精化建议：LLM 分析本体结构，给出重命名/类型精化/层次改进建议并自动应用**
- 系统与模型配置查看

系统由三部分组成：

- 前端：Next.js 16 + React 19 + TypeScript
- 后端：FastAPI + SQLite + OpenAI SDK
- 语义引擎：Ontop CLI 5.5.0 + Ontop Endpoint

## 2. 目录说明

核心代码集中在以下目录：

- `ontop-ui/backend`：FastAPI 后端、SQLite、Ontop 调用、AI 接口
- `ontop-ui/frontend`：前端页面、API 代理、可视化界面
- `ontop-ui/docker`：前后端与 PostgreSQL 的 Docker 构建资源
- `ontop-ui/docs`：说明文档、截图、测试计划

关键入口文件：

- `backend/main.py`：后端入口
- `backend/config.py`：运行配置
- `frontend/src/server.ts`：前端 Node 服务入口，同时代理 `/api/*`
- `docker-compose.yml`：默认 retail 演示环境
- `docker-compose.lvfa.yml`：LVFA 演示环境

## 5. 环境变量

后端主要环境变量定义于 `backend/config.py`。

### 5.1 Ontop 相关

- `ONTOP_BASE`：Ontop 测试目录根路径
- `ONTOP_CLI`：Ontop CLI 可执行文件路径
- `ONTOP_OUTPUT`：输出目录
- `ONTOLOGY_FILE`：默认本体文件路径
- `MAPPING_FILE`：默认映射文件路径
- `PROPERTIES_FILE`：默认 JDBC 属性文件路径

默认本地值会指向项目中的：

- `./docker/ontop-cli/ontop`
- `./docker/backend/ontop-output`

### 5.2 服务相关

- `ONTOP_ENDPOINT_URL`：默认 `http://localhost:8080`
- `ONTOP_ENDPOINT_PORT`：默认 `8080`
- `FASTAPI_PORT`：默认 `8000`

### 5.3 LLM 相关

- `LLM_BASE_URL`：默认 `http://localhost:1234/v1`
- `LLM_API_KEY`：默认 `lm-studio`
- `LLM_MODEL`：默认 `zai-org/glm-4.7-flash`

### 5.4 安全相关

- `ENCRYPTION_KEY`：可选，用于覆盖 SQLite 中敏感字段加密密钥

如果不提供，系统会自动在 `backend/data/.encryption_key` 生成 Fernet 密钥。

功能：

#### 7.4.1 表结构 Tab（默认）

- 选择一个或多个表
- 自动补齐依赖表
- 查看预估类与对象属性
- 生成版本化输出目录

后端逻辑说明：

- 如果 `mode=partial` 或传入 `tables`，则执行局部 Bootstrap
- 局部 Bootstrap 会先跑一次全量 Ontop Bootstrap
- 再根据选定表过滤映射，并重新生成裁剪后的本体

适用场景：

- 只想针对局部业务域生成本体
- 先做小范围建模验证

#### 7.4.2 语义建模 Tab

语义建模 Tab 在 Bootstrap 之前提供一个语义确认环节，让用户看清"表→类、列→属性、FK→关系"的推断结果。

**使用流程：**

1. 在左侧表列表中勾选需要建模的表（支持多选）
2. 点击右侧"语义建模" Tab 切换到语义视图
3. 点击"语义分析"按钮，系统自动推断候选
4. 查看三组候选结果：
   - **类候选**：每张表对应一个 OWL Class，类名由表名转 PascalCase 生成
   - **数据属性候选**：每列对应一个 DataProperty，包含 XSD 类型推断和语义标记
   - **对象属性候选**：外键对应 ObjectProperty，连接源表与目标表
5. 确认无误后点击"生成本体"，系统生成本体文件和映射文件
6. 查看生成结果（版本号、ontology_path、mapping_path、manifest_path）

**推断规则说明：**

| 输入 | 推断规则 | 输出示例 |
|------|---------|---------|
| 表名 | 去除前缀，转 PascalCase | `account` → `Account` |
| 主键列 | 标记为标识符属性 | `account_id` → `identifier (PK)` |
| 含 name/title/code 列 | 标记为 label 候选 | `legal_name` → `legalName (label)` |
| 含 created_at 等系统列 | 标记为 system，降权 | `created_at` → `createdAt (system)` |
| 普通列 | 转 camelCase + XSD 类型映射 | `balance` → `balance (xsd:decimal)` |
| 外键（目标表在选中集合内） | 生成对象属性，status=accepted | `account.global_id → customer` → `hasCustomer` |
| 外键（目标表不在选中集合内） | 生成外部引用，status=external | `account.project_id → property_project` → `refPropertyProject` |

**候选状态说明：**

| 状态 | 含义 | 颜色 |
|------|------|------|
| accepted | 自动推断通过，默认接受 | 蓝色 |
| system | 系统列（created_at 等），降权处理 | 灰色 |
| external | 目标表不在选中集合内，需补充 | 白色描边 |

**适用场景：**

- 在 Bootstrap 之前预览语义推断结果
- 确认表名到类名的映射是否合理
- 确认列到属性的映射和 XSD 类型是否正确
- 确认外键关系是否需要补充依赖表
- 对局部业务域做语义建模验证

功能：

### 7.7 AI 助手 `/ai-assistant`

功能：

- 输入自然语言问题
- SSE 流式展示执行步骤
- 查看生成出的 SPARQL、SQL 和结果文本
- 最终生成中文答案

后端执行链路：

1. 读取当前映射文件，抽取类、属性、关系与前缀
2. 根据系统提示词构造 SPARQL 生成提示
3. 调用 LLM 生成 SPARQL
4. 自动补全缺失的 `PREFIX`
5. 调用 Ontop Endpoint 获取 SQL 改写与查询结果
6. 再调用 LLM 生成最终自然语言回答

流式事件类型包括：

- `analyzing`
- `sparql_generated`
- `executing`
- `executed`
- `answer`

注意：

- 当前 AI 页面里的快捷问题是页面内置常量
- 设置页保存的“快捷问题”当前没有接入此页面读取逻辑

### 7.9 语义注释 `/annotations` ⭐ 新增

**核心时序**

```
Bootstrap
  ↓
  raw_ontology.ttl（纯结构，不拉盔）
  ↓ LLM 自动标注（异步）
  semantic_annotations 表（status=pending）
  ↓ 人工审核
  status=accepted / rejected
  ↓ 点击「合并到本体」
  active_ontology.ttl（raw + accepted 标注）
```

**主要功能**

- **统计卡片**：展示待审核 / 已接受 / 已拒绝 / 合计 历史数量
- **审核 Tab**：
  - **待审核**：LLM 生成的标注，可逐条接受✓ / 拒绝✗ / 编辑✏
  - **已接受**：将合并入 active TTL，可退回待审 ↩
  - **已拒绝**：不写入 TTL，可重新审核 ↩
- **激活过滤**：搜索实体名 / 按类型过滤（类/数据属性/对象属性）
- **批量操作栏**：全选 / 全部接受 / 全部拒绝
- **编辑对话框**：修改中文名称、中文说明、英文名称，保存后 source=human、自动 accepted
- **「合并到本体」按鈕**：触发 POST /annotations/{ds_id}/merge

**人工标注优先级规则**

| 操作 | source | status | Bootstrap 重跑后 |
|------|--------|--------|----------------|
| LLM 自动生成 | llm | pending | 被替换 |
| 用户点✓接受 | llm | accepted | 永久保留 |
| 用户手动编辑 | human | accepted | 永久保留，不被 LLM 覆盖 |
| 用户点✗拒绝 | llm | rejected | 永久保留 |

**合并策略**

- raw TTL 内容原样保留，不修改任何已有三元组
- accepted 注释以 Turtle 注释块追加到 output_path
- 已在 raw TTL 中有 rdfs:label 的实体不重复写入
- active TTL 路径：`DATA_DIR/{ds_id}/active/merged_ontology.ttl`

**对 AI 查询的影响**

`/ai/ontology-summary` 接口优先从注释库读取 accepted 的标注，将其作为 class_labels
传入 LLM 生成 SPARQL 的 Prompt，显著提升自然语言转 SPARQL 的准确度。

### 7.10 业务词汇表 `/glossary` ⭐ 新增

**功能**

将业务口语词（如"欠款"、"物业费"）显式映射到本体属性/类（如 `bill#balance_overdue`），
在 AI 查询时根据用户问题关键词自动筛选注入 Prompt，消除 LLM 猜测属性名导致的查询失败。

**注入机制（Top-K 匹配）**

用户提问时，系统对所有词汇表条目按关键词命中打分（主词=3分、别名=2分、URI=1分），
选取得分最高的前 12 条注入 SPARQL 生成模板。问题无命中时，注入前 12 条兜底。

**主要功能**

- **统计卡片**：全部 / LLM 生成 / 人工添加 数量
- **词汇表格**：展示词汇主词、别名、映射属性/类 URI、来源（LLM/人工）
- **「自动生成」按钮**：从已 accepted 注释层调用 LLM 推导业务词汇（后台任务）
- **新增 / 编辑对话框**：手工维护 term、entity_uri、entity_kind、别名、描述、示例问法
- **导出 JSON**：导出词汇表供跨数据源复用或备份
- **模糊搜索 + 类型过滤**

**人工词汇优先规则**

| 来源 | 被自动生成覆盖? |
|------|----------------|
| LLM  | 是（每次重新生成时覆盖） |
| 人工（source=human） | **否**，永久保留 |

**全局词汇**

`ds_id=''` 的词汇条目为全局词汇，自动合并到每个数据源查询时的注入上下文。

### 7.11 端点注册表（后台功能）

**功能**

Bootstrap 完成后自动将数据源的本体/映射/属性文件路径注册到 `endpoint_registry` 表。
用户可通过 API 切换"激活数据源"，后端将文件同步到共享 active 目录并触发 Ontop reload。

**切换流程（约 5-10 秒）**

```
PUT /api/v1/endpoint-registry/{ds_id}/activate
  ↓ 文件同步到共享 active 目录
  ↓ 调用 Ontop Admin API reload（或 fallback 重启端点进程）
  ↓ 更新 is_current 标记
```

> 切换期间的 SPARQL 查询会返回 503，重启完成后自动恢复。

**当前限制**

同一时刻只有一个 SPARQL 端点在服务（路径1 轻量实现）。
真正并发多数据源需路径2：每个数据源独立容器（规划中）。

### 7.12 本体精化建议 `/refinement` ⭐ 新增

**功能**

Bootstrap 生成的本体往往带有数据库命名风格（表名前缀、冗余字段名称等）。
精化建议模块通过 LLM 分析本体结构和已审核注释，给出结构化改进建议并支持一键应用。

**建议类型与风险分级**

| 类型 | 含义 | 自动应用 |
|------|------|----------|
| `RENAME_CLASS` | 重命名类（去掉表名前缀等） | ✅ 自动 |
| `RENAME_PROPERTY` | 重命名属性（去掉冗余前缀） | ✅ 自动 |
| `REFINE_TYPE` | 精化 XSD 数据类型（string→decimal） | ✅ 自动 |
| `ADD_LABEL` | 补充缺失中文标注（写入注释层） | ✅ 自动 |
| `ADD_SUBCLASS` | 添加子类关系（需确认层次合理性） | 👤 人工 |
| `MERGE_CLASS` | 合并冗余类（高风险） | 👤 人工 |

**操作流程**

1. 在 `/refinement` 页选择数据源，点击「AI 分析」（约 30s 后台生成）
2. 建议列表按优先级（高/中/低）排序展示，红边框 = 高优先级
3. 逐条点击**接受 ✓** / **拒绝 ✗**，或展开查看理由与人工操作指引
4. 点击**批量应用**，将所有已接受 + auto_apply=true 的建议写入 TTL
5. TTL 修改前自动备份为 `.ttl.bak`

**自动应用实现**

- `RENAME_CLASS/PROPERTY`：正则替换 URI local name，词边界保护避免误替换
- `REFINE_TYPE`：替换 `rdfs:range` 中的 XSD 类型声明
- `ADD_LABEL`：写入语义注释层（status=pending，需前往 `/annotations` 审核后合并）

## 8. 推荐使用流程

建议按以下顺序使用：

1. 进入 `/datasource` 创建并测试数据源
2. 进入 `/db-schema` 查看结构，勾选目标表
3. 在 `/db-schema` 的“语义建模” Tab 点击“语义分析”，确认推断结果
4. 点击“生成本体”生成本体和映射文件
5. 在 `/mapping` 检查和修正自动生成的 OBDA 规则
6. 在 `/mapping` 中验证映射并重启端点
7. 在 `/sparql` 手工验证查询
8. 在 `/settings` 配置好 LLM
9. 🆕 进入 `/annotations` 审核 LLM 自动生成的语义标注
10. 🆕 批量接受标注后点击「合并到本体」
11. 🆕 进入 `/glossary` 点击「自动生成」生成业务词汇表，或手工添加关键词汇
12. 🆕 进入 `/refinement` 点击「AI 分析」，审核并应用本体精化建议
13. 在 `/ai-assistant` 做自然语言问答（AI 已能利用语义注释 + 业务词汇表）
14. 在 `/ontology` 查看精化后的图谱和本体定义
15. 🆕 如需切换数据源：`PUT /api/v1/endpoint-registry/{ds_id}/activate`

## 11. 已知限制

以下内容来自当前代码实际状态：

- 登录页不是正式认证，仅为演示入口
- 系统页展示的用户信息是 mock 数据
- AI 助手页的快捷问题未接入设置页保存结果
- 数据源页只支持全量 Bootstrap；局部 Bootstrap 在数据库概览页执行
- Ontop Endpoint 需要显式启动或重启后，SPARQL/AI 查询才可正常工作
- 前端开发服务器重启后才会识别新路由（如 `/annotations`、`/glossary`、`/refinement`）
- 注释审核 UI 尚无人工优化建议（只显示文字标注，无业务指导策略）
- **端点切换（路径1）**：同一时刻只有一个 SPARQL 端点在服务，切换约需 5-10 秒重启窗口期；期间 SPARQL 请求返回 503
- **TopBar 数据源切换器**尚未实现（规划中），目前须通过 `PUT /api/v1/endpoint-registry/{ds_id}/activate` API 切换激活数据源
- **本体精化自动应用**使用正则替换，修改前会创建 `.ttl.bak` 备份；极少数情况下可能误替换名称相同的子字符串
- `ADD_SUBCLASS` 类建议不支持自动应用，需人工追加 Turtle 三元组到本体文件
- **REFINE_TYPE** 自动应用时，若 TTL 中属性无 `rdfs:range` 声明，会自动追加一行；若有则替换
- **LLM JSON 输出不稳定**：中文标签可能导致未闭合引号，内置 repair 机制（逐块提取+修复）可兜底恢复
- **API Key 鉴权**：前端通过 `X-Internal-Request` 头自动放行；外部访问需携带 `X-API-Key` 头或 `?api_key=` 参数

### 7.14 API Key 鉴权

系统通过中间件保护 `/api/v1/*` 和 `/mcp/*` 端点。

**鉴权规则（按顺序判断）：**

1. `api_enabled=false`（publishing 页未启用 API） → 放行
2. 请求携带 `X-Internal-Request` 头 → 放行（前端自动附加）
3. 来源 IP 为 `127.0.0.1` 或 `::1` → 放行
4. 检查 `X-API-Key` 请求头或 `api_key` 查询参数 → 匹配则放行
5. 以上均不满足 → 返回 401

**使用方式：**

```bash
# 外部调用示例
curl -H "X-API-Key: your-api-key" \
  http://your-server:8001/api/v1/sparql/query \
  -d '{"query": "SELECT ?s WHERE { ?s a <http://example.com/Class> } LIMIT 10"}'
```
