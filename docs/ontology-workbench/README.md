# Ontology Workbench 设计说明

## 页面定位

`Ontology Workbench` 是一个独立的新页面，用于把关系数据库结构半自动提升为本体模型与映射规则。

这个页面不替代现有页面，而是作为统一入口：

- 上游连接数据库 schema
- 中间完成语义建模
- 下游产出 ontology draft 与 mapping preview

现有页面继续保留：

- `db-schema` 负责查看数据库结构
- `mapping` 负责编辑 OBDA 映射
- `ontology` 负责查看本体与图谱

## 设计目标

页面解决的核心问题不是“如何手写映射语法”，而是：

- 从原始表结构识别业务实体
- 把表、列、外键提升为类、属性、关系
- 让用户确认和修正语义
- 自动生成 ontology 与 mapping

一句话定义：

> 把“数据库结构”翻译成“语义模型”，再把“语义模型”落成“本体 + 映射”。

## 核心原则

### 1. 先语义，后语法

先让用户确认“它是什么”，再生成“怎么写映射”。

### 2. 先候选，后编辑

系统先推荐类、属性、关系，用户主要做确认和修正，而不是从空白开始。

### 3. 同屏闭环

同一页内同时看到：

- 来源表结构
- 语义建模结果
- 本体草稿
- 映射预览

### 4. 默认可用，逐步深入

- 初级用户：卡片确认即可完成
- 高级用户：展开查看 URI、映射规则、TTL、约束

## 页面模块

### 1. HeaderBar

顶部全局操作区：

- 页面标题
- 数据源选择
- 自动识别
- 保存草稿
- 生成本体
- 生成映射
- 发布

### 2. ProgressStepper

流程步骤：

- 选择数据源
- 识别结构
- 语义建模
- 映射预览
- 验证发布

### 3. SchemaExplorerPanel

左栏，回答“数据库里有什么”：

- 表列表
- 表搜索
- 字段列表
- 主键 / 外键 / 唯一约束
- 表之间关系
- 样例值

### 4. SemanticModelingPanel

中栏，回答“这些结构在语义上是什么”：

- 表到类候选
- 列到数据属性候选
- 外键到对象属性候选
- 关系表 / 桥表 / 枚举表识别
- 改名、合并、忽略、切换类型

### 5. OntologyDraftPanel

右栏，回答“当前会形成什么本体”：

- 类清单
- 属性清单
- 图谱关系
- IRI 模板
- domain / range
- 约束提示

### 6. BottomPreviewDrawer

底部抽屉，回答“最终会生成什么”：

- Mapping Preview
- RDF Preview
- Validation
- Generated TTL

## 核心交互流

```text
选择数据源
→ 读取 schema
→ 自动生成类/属性/关系候选
→ 用户在中栏确认和修改
→ 右栏实时刷新 ontology draft
→ 底部实时刷新 mapping / RDF preview
→ 点击生成并保存
```

## 推荐的自动推断规则

第一版采用稳定规则，不追求过度智能：

- 表名 -> 类名
- 主键 -> 实体标识
- 普通列 -> datatype property
- 外键 -> object property
- 桥接表 -> 多对多关系
- 码表 / 维表 -> 枚举类或概念类
- 系统字段如 `created_at` / `updated_at` 默认降权
- `name` / `title` / `code` 优先作为标签属性

## 页面必须支持的确认动作

对候选类、属性、关系统一支持：

- 接受
- 改名
- 合并
- 忽略
- 切换类型
- 标记为枚举
- 标记为中间关系

## MVP 范围

第一版只做最小闭环：

- 读取 schema
- 自动推断候选类、属性、关系
- 卡片式确认
- 本体草稿预览
- Mapping Preview
- 基础 Validation

第一版暂不做：

- 复杂拖拽编排
- 图谱双向编辑
- 全量 SHACL 自动生成
- 高级批量规则编辑

## 页面价值

相比直接编辑映射规则，这个页面的价值在于：

- 降低本体构建门槛
- 把数据库结构和语义建模串成一个闭环
- 让用户先理解“语义”，再处理“规则”
- 为后续 ontology、mapping、validation 页面提供统一入口
